AI Interview Proof Map
The most probable interview questions for AI Solutions Architect and AI Solutions Engineer roles — mapped directly to how TSM Shell already answers them. Every answer is a live proof point, not a rehearsed story.

CATEGORY 1 · ARCHITECTURE & SYSTEM DESIGN
Q: "How would you design a multi-agent AI system for enterprise use?" TIER 1 · MOST ASKED
TSM ANSWER: TSM Shell runs a 4-stage multi-agent chain across 7 verticals: War Room (intake agent) → Strategist (analysis agent) → Executive Portal (decision agent) → Explainability Layer (audit agent). Each agent has a defined role, passes structured context to the next, and cannot skip a stage. In Microsoft terms: multi-agent orchestration with A2A handoff, RAG grounding at the Strategist layer, and HITL authorization at the Executive Portal before any action executes.

PROOF POINT: HC Denial War Room → HC Strategist → HC Executive Portal → BNCA Explainability. Live at tsm-shell.fly.dev. Replicated across all 7 verticals.
Q: "How do you handle context passing between AI agents?" TIER 1
TSM ANSWER: TSM uses a relay key pattern — each stage receives a structured payload from the prior agent via server-side proxy endpoints. No client-side key exposure. Context accumulates through the chain: the War Room passes the classified denial type, the Strategist receives it as input and appends its recommendation, and the Exec Portal receives the full chain state for HITL sign-off. This is identical to A2A protocol in Microsoft Copilot Studio and AWS Step Functions.

PROOF POINT: /api/war-room/stream → /api/hc/stream → relay key pattern in server.js. Groq routed server-side, never client-exposed.
Q: "Walk me through how you'd architect an AI solution from intake to decision." TIER 1
TSM ANSWER: UNDERSTAND → DECIDE → EXECUTE → TRUST. The BPO chain is the canonical example: Situation Room (understand the operational problem), Strategist V2 (model 3 decision paths with confidence scores), Exec Portal (human approves the path), Explainability Panel (audit trail with reasoning chain, evidence sources, and business impact delta). No stage is optional. The candidate — or the enterprise user — cannot skip EXECUTE without completing UNDERSTAND.

PROOF POINT: BPO Situation Room → BPO Strategist V2 → BPO Executive Portal. All 7 verticals share this architecture with vertical-specific payloads.
CATEGORY 2 · GOVERNANCE & RESPONSIBLE AI
Q: "How do you implement Human-in-the-Loop in a production AI system?" TIER 1 · MOST ASKED
TSM ANSWER: Every Executive Portal in TSM Shell has a HITL modal — a named, timestamped, auditable authorization gate. The AI recommendation is surfaced with a confidence score, evidence sources, and reasoning chain. The human reviewer can approve, override with a reason code, or escalate. No action executes without this sign-off. The approval record is logged with digital signature and stored permanently. This is not a UI feature — it's an architectural requirement baked into every vertical.

PROOF POINT: HITL Approval Center (this platform), Executive Portals across HC, FinOps, Insurance, RE, BPO, Legal, Construction verticals. All 7 implement the same pattern.
Q: "How do you make AI decisions explainable to non-technical stakeholders?" TIER 1
TSM ANSWER: TSM uses a structured explainability contract per vertical: every AI output includes a confidence score (0–100%), evidence sources (documents processed, payer rules, precedents), a reasoning chain ("AI flagged CO-50 because…"), and a business impact delta ("resolving this denial recovers $4,200"). The output format is ===BRIEF=== for the human and ===JSON=== for downstream systems. Staffing firms can hand the JSON export to enterprise clients as a placement credential.

PROOF POINT: .tsm-xp CSS class, ParseExplain/RenderExplain functions in all 7 war room files. Governance panel in this platform.
Q: "How do you prevent sensitive data from being exposed to the AI model?" TIER 2
TSM ANSWER: All Groq API calls in TSM Shell are routed through server-side proxy endpoints. The API key never reaches the client. Session auth uses tsm-auth.js middleware with 8-hour HttpOnly cookies — all /html/* routes are gated. Secrets are stored as Fly.io environment variables (TSM_ACCESS_CODE, TSM_SESSION_SECRET). The client only ever sees the AI response, never the key or the raw system prompt.

PROOF POINT: server.js proxy routes, tsm-auth.js session middleware, Fly.io secrets configuration. Full security audit completed across all 7 verticals.
CATEGORY 3 · ENTERPRISE DEPLOYMENT & INTEGRATION
Q: "How do you deploy and version an AI application in production?" TIER 1
TSM ANSWER: TSM Shell uses a GitHub Actions CI/CD pipeline with auto-deploy to Fly.io on every push to main. The repo is structured with vertical-specific subdirectories, shared middleware, and a centralized server.js that handles all routing. Deployment is containerized, zero-downtime, with health checks. In Microsoft terms: this is Azure DevOps ALM with blue/green deployment to a managed container environment. Every release is an artifact with a traceable commit.

PROOF POINT: .github/workflows/ in Torreyw94/tsm-shell, Fly.io deployment at tsm-shell.fly.dev, tsm-deploy.sh.
Q: "How do you scale AI solutions across multiple business verticals?" TIER 2
TSM ANSWER: TSM Shell uses a shared architecture pattern — one server, one auth layer, one relay bus pattern — deployed across 7 verticals. Each vertical gets its own war room, strategist, and exec portal, but they share the same HITL governance pattern, explainability contract format, and relay key convention. Adding a new vertical means building one new HTML chain that plugs into existing infrastructure. The collective intelligence layer (tsm-collective-bnca.html) cross-references anomalies across all verticals simultaneously.

PROOF POINT: HC, FinOps, Insurance, RE, BPO, Legal, Construction — 7 independent verticals, one deployment, shared middleware.
Q: "How do you handle RAG — Retrieval-Augmented Generation — in your architecture?" TIER 1
TSM ANSWER: TSM's memory engine (tsm-memory-engine.js) functions as the cross-upload insight layer — documents uploaded through tsm-doc-search-multi.html are classified by sector and routed to the appropriate vertical's grounding context before the AI generates its response. The system retrieves relevant prior decisions, payer rules, and anomaly definitions to ground the output. This is the RAG pattern — retrieve first, generate second — implemented without a vector database by using structured document classification and session context.

PROOF POINT: /html/shared/tsm-memory-engine.js, tsm-doc-search-multi.html multi-vertical document intake hub.
CATEGORY 4 · PRESALES & SOLUTIONS ARCHITECTURE
Q: "How do you scope an AI solution for a client who doesn't know what they need?" TIER 1 · PRESALES
TSM ANSWER: Start with the decision, not the technology. Every TSM vertical starts by identifying what decision the human is trying to make (deny or appeal? approve or escalate? place or revise?) and works backward to: what data exists, what the AI can surface, and where the human approval gate belongs. The discovery conversation follows: What's your highest-cost manual decision? What's the error rate? What happens when it's wrong? That scopes the solution — the technology selection comes last.

PROOF POINT: The BPO Situation Room was built to formalize exactly this scoping process. UNDERSTAND before DECIDE is architectural, not cosmetic.
Q: "What's the difference between an AI agent and a workflow automation?" TIER 2 · AB-100
TSM ANSWER: Workflow automation executes a fixed sequence of steps based on rules — it doesn't reason. An AI agent perceives inputs, reasons about them in context, and selects actions toward a goal. TSM's war rooms are agents: they receive unstructured input (a denial code, a doc anomaly, a supplier disruption), reason about it against a knowledge base, and recommend a path — which a human then approves. The Honeywell industrial war rooms (plant incident, supplier shutdown) are the clearest examples of agentic reasoning vs. rule-based triggers.

PROOF POINT: plant-incident.html, supplier-shutdown.html — Honeywell vertical war rooms. Agentic response vs. rule-based trigger comparison built in.
Q: "Tell me about a time you built something with AI end-to-end." TIER 1 · BEHAVIORAL
TSM ANSWER: Built TSM Shell — a 7-vertical AI decision intelligence platform — solo, deployed to production, handling real enterprise workflows. Started with a healthcare RCM denial war room and expanded to FinOps, Insurance, RE/MLO, BPO, Legal, and Construction. Every vertical has its own AI chain (UNDERSTAND → DECIDE → EXECUTE → TRUST), HITL governance gate, explainability contract, and audit trail. Server-side Groq inference, session auth middleware, GitHub CI/CD to Fly.io. It's not a portfolio project — it's a live platform at tsm-shell.fly.dev that staffing firms can demo to enterprise clients today.

PROOF POINT: tsm-shell.fly.dev — open a war room, run a scenario, show the explainability panel, point to the GitHub Actions pipeline. The demo is the answer.
